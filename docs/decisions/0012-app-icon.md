# ADR 0012 — App icon: SVG at the root, generated .ico committed

**Status:** accepted · **Date:** 2026-07-12

## Context

`icon.svg` (a typewriter, 512×512) is the app's identity. Windows wants an `.ico` for the
executable's Win32 resource, and Avalonia wants a loadable resource for the window/taskbar
icon at runtime. Neither consumes SVG.

## Decision

- **`icon.svg` at the repo root is the source of truth.**
- The generated `src/Steno.App/Assets/icon.ico` (16/24/32/48/64/128/256 px, PNG-compressed
  entries) and `icon.png` (256 px) are **committed**, not generated at build time.
- `<ApplicationIcon>` bakes the `.ico` into the exe; `<AvaloniaResource Include="Assets\**" />`
  ships it for `Window.Icon="/Assets/icon.ico"`.

## Why commit the generated files

Rasterising SVG needs a renderer, and there is none in the .NET SDK: doing it at build time
would drag a Skia/ImageMagick dependency (or an external tool) into every build, on every
machine, forever — to produce a file that changes roughly never. The icon was rendered once
with Svg.Skia in a throwaway project.

**If `icon.svg` changes, the `.ico` must be regenerated by hand.** That is the cost, and it is
the right trade for a file that changes once a year. The build does not check they are in sync.

## Note

Every `.ico` entry is PNG-compressed, including the small sizes. Windows 10/11, Explorer and
Avalonia all accept this; only pre-Vista shells require BMP entries, and they are not a target.
